# 5.9 Deployment & runtime constraints

§§5.1–5.8 each committed an algorithmic stage and a runtime-budget row. This section is where those rows have to add up. It reconciles the eight source sections into one chapter-level picture, calls out the deployment invariants that bind every stage at once (TF freshness, sensor class, scheduler tail behaviour), and gives the reader the framing they need before they meet [[6_0_overview_EN|Ch 6]]: what classical LiDAR detection still ships in production, where it has been displaced, and what survives inside the stacks that displaced it.

The section is a synthesis layer, not a ninth algorithm. The useful output is a bounded claim, a runtime budget interpretation, and a small catalog of chapter-wide failure modes that no single upstream section can own alone.

## The bounded industry claim

The chapter's honest assessment is narrower than a slogan but firmer than a hedge. It rests on three definitions that have to stay tight whenever the claim is restated.

- **"Production stacks"** — stacks deployed on public roads in passenger vehicles or revenue-service robotaxi. Test fleets, demo videos, and research benchmarks do not count.
- **"Primary detection"** — the step that outputs class, 3-D bounding box, heading, and velocity to the planner. This explicitly excludes AEB-style emergency obstacle detectors, generic obstacle / occupancy fallbacks, and redundancy / sanity-check monitors, all of which are often classical and remain so by design.
- **"High-speed open-road AD"** — highway and urban-arterial driving at speeds at or above 30 km/h. Low-speed and restricted-ODD systems are a separate bucket with their own rules ([[1_10_odd_primer_EN|Ch 1 §1.10]]).

With those definitions in place: in 2026, publicly disclosed stack architectures for high-speed open-road robotaxi / L4 consistently use DL-based primary 3-D bounding-box prediction, with Waymo Open Dataset stack disclosures, Apollo public source, and Pony.ai technical reporting as representative public evidence as of 2024–2026. The original informal phrasing "universally DL-based" is useful only inside that disclosed-architecture boundary; it should not be read as a claim that every proprietary stack has published enough detail to audit every production module. For consumer NOA with LiDAR (XPeng, Nio, Li-Auto class), teardown reports, supplier disclosures, patents, job postings, conference talks, official architecture posts, and observed behavior point in the same direction, but the evidence is inferential rather than complete. This section therefore says public evidence and industry practice **strongly indicate** DL-based primary bbox prediction for consumer NOA, not that the book can prove the internals of every closed vehicle release.

The bounded claim is about the **primary bbox predictor role**. Production stacks blur the learned / classical boundary elsewhere: learned segmentation can feed classical clustering or tracking; rule-based post-processing can wrap neural detections; learned occupancy can replace bbox primaries in some planning paths; and map / occupancy planning can reduce the centrality of bbox outputs altogether. Those interfaces matter, but they do not change the narrower role definition above. If the module whose output the planner treats as the main `(class, 3-D box, heading, velocity)` obstacle stream is a learned detector, the stack is not a production stack running primary classical detection, even when half the surrounding machinery is classical.

Where pure-classical primary detection still ships today:

- Low-speed and restricted-ODD AD: ports, mines, warehouses, last-mile sidewalk robots, airport tugs.
- AEB-style safety modules and emergency-braking obstacle detectors in production passenger vehicles, where determinism and ASIL-rateable behaviour outweigh detection rate.
- Curb / barrier / free-space monitors and redundancy / sanity-check layers in DL-primary stacks.
- Off-highway autonomy (agriculture, construction) where ODD is constrained and certification windows are different.

### Worked Example — applying the bounded claim

Suppose Stack X publishes a DL LiDAR detector called `Y` that emits class labels, 3-D boxes, headings, velocities, and confidence scores at 10 Hz. Downstream, it uses a classical Kalman / IMM tracker with Hungarian association, a Patchwork++ ground-removal monitor, and an occupancy-grid Generic Obstacle Detection fallback. Does this count as "production stacks running primary classical detection"? No. Under this section's definition, primary detection is the bbox predictor role, and `Y` owns that role. The Kalman tracker is load-bearing and classical, but it consumes detections; it does not decide the primary per-frame bbox hypotheses from points. The ground segmenter and occupancy fallback are also classical, but they are front-end, monitor, or fallback paths.

Now change the example. Stack Z runs on an airport tug at low speed inside a fenced ODD. It uses deskew, Patchwork++, Euclidean clustering, L-shape fitting, and Kalman tracking as the only obstacle stream the planner consumes. That is a production deployment running classical primary detection, but it sits in the low-speed / restricted-ODD bucket, not the high-speed open-road AD bucket. The distinction is not semantic. The same algorithm that is reasonable for a tug at 12 km/h inside a mapped apron is not evidence that high-speed robotaxi / consumer-NOA primary detection still ships as pure classical clustering plus fitting.

## Load-bearing classical pieces inside DL-primary stacks

The same production stacks that displaced classical primary detection still depend on classical pieces elsewhere in the pipeline. Five survive intact, and each is taught in this chapter.

1. **Preprocessing** — deskew, outlier removal, voxel downsampling, ROI gating. The DL detector still consumes a cloud whose geometry has been made physically meaningful: §5.1's per-point `time` convention, sweep-end stamp, `lidar → base_link` transform, SOR/ROR filtering, height clipping, and `pcl::VoxelGrid` leaf-size decision are not automatically learned away. A PointPillars- or CenterPoint-style head can be learned, but its input tensor still inherits the sensor timing, extrinsics, voxel-size, accumulation, and point-count choices that §5.1 makes explicit. If deskew fails, Ch 6 sees a doubled car, not a neural-network-only problem.
2. **Ground segmentation as a CNN front-end** — running Patchwork or a Himmelsbach-style radial segmenter before the learned detector can materially reduce downstream input size. [[5_2_ground_segmentation_EN|§5.2]] is often the classical version in front of an end-to-end-trainable network, since it can cut input cloud size meaningfully before the learned detector runs. The deployment-safe phrasing is qualitative because the reduction depends on sensor class, ODD, voxel leaf size, whether filtering happens before or after learned voxelization, and whether the ground branch preserves curbs and low obstacles conservatively. §5.2's stronger lesson is not a universal percentage; it is that early ground deletion is lossy and must be validated against curb, ramp, spray, and elevated-flat-object failures.
3. **Tracking** — Kalman / IMM / Hungarian / JPDA on top of DL detections. The temporal layer over the learned detector is still very often the lifecycle, gating, covariance, and motion-model machinery from [[5_5_classical_tracking_EN|§5.5]], with learned ID embeddings or appearance cues as optional add-ons. The important inheritance is the frame policy: tracks live in `odom`, measurements are transformed at their detection stamps, and yaw updates are treated as weak when §5.4-style shape evidence is ambiguous. A learned detector changes the measurement source; it does not remove the need for association, lifecycle, out-of-sequence handling, coasting policy, and covariance diagnostics.
4. **Generic Obstacle Detection / occupancy** as the safety fallback for unknown classes. A DL detector can only detect the classes it was trained on; the class-agnostic occupancy fallback from [[5_7_occupancy_freespace_map_roi_EN|§5.7]] is what catches the overturned truck, the fallen mattress, and the road-debris pile. §5.7 owns the implementation depth (log-odds, ray carving, age awareness, blob extraction, planner arbitration); the deployment-survival point here is that this fallback must stay active alongside the DL primary detector so that unknown-class actors still surface to the planner.
5. **HD-map ROI gating** — Apollo HDMap polygon ROI lookup and Autoware `compare_map_segmentation`. Both pieces from [[5_7_occupancy_freespace_map_roi_EN|§5.7]] still gate the input to the DL detector and contribute the map-subtraction residual, with [[5_6_registration_EN|§5.6]] Role 2 (map subtraction) and Role 4 (map-aided ROI consistency) keeping the alignment honest. The inheritance from §5.6 is cadence-sensitive: scan-to-map GICP for map subtraction is too expensive to pretend it is a cheap every-frame filter, while the ROI mask can be hot-path only if the map-frame transform and tile lookup are fresh.

Classical clustering and shape fitting from §§5.3–5.4 also survive, but usually as fallback, regression baseline, cleanup branch, or generic-obstacle geometry rather than as the high-speed primary proposal path. That is why the five-item list above names the pieces that remain load-bearing in DL-primary stacks, not every classical module taught in Ch 5.

## China-vs-US deployment priors

China's consumer ADAS market embraced LiDAR early — XPeng's 2021 partnership with Livox brought solid-state LiDAR onto a production passenger vehicle, and Nio's Aquila-style sensor-stack disclosures made roofline LiDAR part of the public consumer-NOA conversation. The US split is sharper: robotaxi programs such as Waymo and Zoox are LiDAR-heavy on the L4 side, while Tesla maintains the explicit camera-only counter-position on the consumer side. The claim "primary detection is DL-based" holds across both regions under the evidence boundary above; the claim "the production stack contains a LiDAR" does not. The camera-only counter-position belongs to [[4_0_overview_EN|Ch 4]], not here.

## Per-stage runtime budget — the deployment consequence

The eight source sections (§§5.1–5.8) reconcile into nine displayed rows because §5.6 has two distinct runtime roles: Role 1 (deskew refinement, point-to-plane ICP) runs every frame, while Role 2 (map subtraction, GICP scan-to-map) runs sub-rate every third frame. The table below is therefore not an arithmetic mistake; it is the chapter making the cadence split visible.

The table is normalized to a 10 Hz mechanical-spinning VLP-32C / HDL-32E single-roof LiDAR on a Jetson-class edge module. Numbers are illustrative — per-deployment latency, memory, and tail behaviour vary with sensor, RMW, allocator, CPU class, and accumulation cadence, and should be measured rather than assumed. The book commits to a PyTorch-training → ONNX/TensorRT-inference → C++ ROS2 node → Jetson-class edge GPU pipeline ([[1_9_deployment_target_EN|Ch 1 §1.9]]); §5.9's illustrative numbers assume that target's CPU/GPU class, and any deviation from it rescales every row below.

| stage | compute | p50 (ms) | p99 (ms) | memory (MB) | cadence |
|---|---|---:|---:|---:|---|
| [[5_1_pointcloud_preprocessing_EN\|§5.1]] preprocessing | cpu | ~6 | ~18 | ~96 | every-frame |
| [[5_2_ground_segmentation_EN\|§5.2]] ground segmentation | cpu | 8 | 22 | 80 | every-frame |
| [[5_3_clustering_EN\|§5.3]] clustering | cpu | 8 | 24 | 96 | every-frame |
| [[5_4_object_shape_fitting_EN\|§5.4]] shape fitting | cpu | ~3 | ~9 | ~24 | every-frame |
| [[5_5_classical_tracking_EN\|§5.5]] tracking | cpu | ~1.5 | ~5.0 | ~16 | every-frame |
| [[5_6_registration_EN\|§5.6]] registration — Role 1 ICP | cpu | ~10 | ~22 | (shared with Role 2) | every-frame |
| [[5_6_registration_EN\|§5.6]] registration — Role 2 GICP | cpu | 20 | 45 | 200 | every 3rd frame |
| [[5_7_occupancy_freespace_map_roi_EN\|§5.7]] occupancy + ROI | cpu (gpu-optional) | 8 | 25 | 220 | every-frame; map-subtraction sub-rate |
| [[5_8_ros2_integration_EN\|§5.8]] ROS2 plumbing | cpu | ~0.6 | ~2.0 | ~8 + ~4.2 shared cloud buffer | every-frame |

**Sensor-class normalization.** Six rows assume VLP-32C / HDL-32E single-roof at ~60–70k points per frame after preprocessing. §5.8's row was committed against HDL-64E at ~130k points per frame because that section measures *transport overhead*, where the higher density is the harder case. Transport latency rescales roughly linearly with point count, so the §5.8 number maps to ~0.3 ms p50 on a VLP-32C; the table above keeps the §5.8 figure as committed and flags the rescale here rather than silently averaging across sensor classes. Multi-LiDAR deployments multiply preprocessing, ground, clustering, and fitting memory by the number of sensors; the prior-map tile and HD-map ROI LUT are shared.

**Cadence reconciliation.** §5.6 plays two roles at two different cadences and the budget arithmetic has to keep them apart. Role 1 (inter-sweep deskew refinement, point-to-plane ICP) runs **every frame** at roughly half the GICP cost, contributing ~10 ms p50. Role 2 (map subtraction for change detection, full GICP scan-to-map) runs at **sub-rate every 3rd frame**, contributing ~20 ms p50 only when it fires. The every-frame chain is therefore `preprocess + ground + cluster + fit + track + occupancy + ROS2 plumbing + §5.6 Role 1` ≈ `6 + 8 + 8 + 3 + 1.5 + 8 + 0.6 + 10 ≈ 45 ms p50`. The Role-2 GICP tick adds ~20 ms p50 on the frame it fires; the rest of the period it adds nothing. Reporting a single 35 ms or 55 ms total for the pipeline is misleading either way, and the deal-loop will surface the elision; the split is the honest figure.

**p99 / tail reconciliation.** The p99 column does not sum cleanly. A naive sum over the every-frame chain is `18 + 22 + 24 + 9 + 5 + 25 + 2 + 22 ≈ 127 ms`, well past the 100 ms 10 Hz frame budget; with the Role-2 GICP tick on its sub-rate frame it crosses 170 ms. Real worst-case is bounded by but not equal to the sum because the per-stage tails are not perfectly correlated — a hard frame for §5.6 ICP often coincides with a hard frame for §5.3 clustering (both stress on dense urban scenes, weather, or sparse far returns), but rarely with a §5.8 RMW jitter spike. The honest implication is operational, not arithmetic: the scheduler must be sized for **tail-correlated stalls**, not just p50. A pipeline tuned to fit p50 inside 100 ms but ignoring the p99 envelope will skip frames under correlated load, and the failure shows up not as latency but as tracker covariance growth, occupancy cells that miss updates, and registration that quietly stops keeping up — which is the `5_9.fm.frame_budget_overrun_p99` entry below.

### Worked Example — one correlated stall

Take a nominal 10 Hz stack with a 100 ms frame period. In an easy suburban frame, the every-frame chain lands near the p50 calculation above: roughly 45 ms, leaving enough margin for executor overhead and diagnostics. Now replay a rainy dense-urban frame with spray near the ego vehicle, sparse far returns, and a sharp ego-yaw transient. §5.2 spends near its p99 budget preserving ambiguous curb and spray residuals; §5.3 spends near its p99 budget because the residual cloud is dense near-field and fragmented far-field; §5.6 Role 1 spends near its p99 budget because point-to-plane ICP is refining a difficult deskew / ego-motion frame. Those three stages alone move from `8 + 8 + 10 = 26 ms` p50 to `22 + 24 + 22 = 68 ms` p99. Add the rest of the every-frame chain at ordinary values and the frame is already near or over the 100 ms deadline; if the Role-2 GICP tick also fires, the scheduler must choose what to defer.

The correct overload behavior is not to let every callback queue and publish late. The sub-rate Role-2 map-subtraction tick should be skipped first; every-frame preprocessing, ground, clustering, tracking, occupancy, and diagnostics should preserve timestamp order and drop stale input according to policy. If the executor instead tries to process everything, the visible failure is indirect: the tracker coasts longer, covariance grows, occupancy cells age past their freshness window, map subtraction consumes a stale transform, and the planner receives a syntactically valid but temporally inconsistent world. This is why §5.9 treats tail-correlated stalls as a deployment failure mode rather than a spreadsheet footnote.

**Every-frame aggregation.**

| chain | included stages | p50 interpretation | p99 interpretation |
|---|---|---:|---:|
| normal every-frame path | §§5.1, 5.2, 5.3, 5.4, 5.5, 5.6 Role 1, 5.7, 5.8 | ~45 ms | naive bound ~127 ms |
| Role-2 tick frame | normal path + §5.6 Role 2 | ~65 ms on that frame | naive bound >170 ms |
| degraded overload policy | normal path, Role 2 skipped, stale queues dropped | keep primary path current | bounded by measured replay policy, not by hope |

**Memory total.** A naive working-set sum across the every-frame stages plus the Role-2 GICP allocation is roughly `96 + 80 + 96 + 24 + 16 + 200 + 220 + 8 + 4.2 ≈ 740 MB` per pipeline instance for a single LiDAR. OctoMap-as-a-3-D-layer is **excluded** from the §5.7 220 MB figure and adds 50–200 MB more if enabled. Multi-LiDAR roughly multiplies the preprocessing, ground, clustering, and fitting components; the prior-map tile and HD-map ROI LUT do not multiply, because they are shared.

**Frame-rate consistency.** Every row above assumes 10 Hz mechanical-spinning, the canonical case the chapter's algorithms were tuned against. 20 Hz solid-state sensors (Hesai AT128, RoboSense M-series) halve the **frame period** — the every-frame chain's deadline tightens from 100 ms to 50 ms, and the ~45 ms p50 figure that fits comfortably inside a 100 ms period now leaves almost no margin. The per-stage *compute* does not automatically halve: it tracks point count, algorithm complexity, and memory bandwidth, none of which the frame rate directly changes. Solid-state sensors *can* deliver fewer points per frame than a 10 Hz spinning unit at the same scan resolution, which is a separate point-count argument that does relax the budget; the frame-rate change alone does not. FMCW sensors with per-point radial velocity simplify dynamic-object handling at the registration and tracking stages but do not change the budget arithmetic. Solid-state and FMCW deployments inherit the framework above and rescale the numbers; they do not change the failure modes.

## ROS2 timing reconciliation

The nine displayed rows (eight source sections, with §5.6 split into Role 1 + Role 2) commit one shared invariant — `≤ 50 ms` `/tf` freshness — and one shared transport pattern. Both are chapter-wide, not per-stage. Composable nodes are ROS2 components loaded into one process so their callbacks share an address space, and intra-process pass-through is the optimization that lets a publisher hand a `std::shared_ptr<PointCloud2>` directly to a same-process subscriber with no copy ([[1_5_ros2_humble_essentials_EN|Ch 1 §1.5]]; [[5_8_ros2_integration_EN|§5.8]]). The algorithmic stages from §§5.1–5.7 are typically composed inside a single perception container, with the registration sub-rate component running as a separate composable node so its longer GICP tick does not block the every-frame chain. TF2 freshness is the chapter-wide invariant: every stage transforms in or out of `lidar / base_link / odom / map` at some point in its callback, and a stale `/tf` lookup is not a §5.1 or §5.5 bug — it is a chapter-wide deployment failure that lands wherever the lookup happens to live. `/tf_static` must be present before activation: ROS2 lifecycle nodes have a managed `inactive → active` transition gated on configuration completeness, used here to ensure the static transforms are published and the parameter set is loaded before perception starts publishing. The 20 Hz solid-state shift tightens the deadline but does not by itself tighten the `/tf` staleness tolerance — that tolerance is set by the perception loop's correctness requirements, not by the sensor's frame period.

The §5.8 cadence / QoS table gives the reconciliation rule. Full-cloud live topics use `SensorDataQoS` or intra-process-compatible equivalents with shallow history so stale clouds are dropped rather than processed late. Smaller outputs such as clusters, objects, tracks, grids, ROI masks, and diagnostics can use reliable bounded queues, but they still carry source stamps and reject stale inputs. Registration roles deserve their own callback group or component because a long optimizer tick must not block TF listener callbacks or the every-frame cloud path. The deployment invariant is that every output should be explainable by `(source stamp, output stamp, TF age, queue depth, dropped-frame count, callback duration)`. Without those diagnostics, a runtime budget row is only a hope.

## Field robustness

Rain, fog, and sensor degradation degrade the classical LiDAR pipeline through related but distinct mechanisms: rain produces near-field spray returns and surface-spec backscatter, fog attenuates and shortens useful range, and sensor faults (window contamination, beam misalignment, channel dropout) reduce return density and stability. Low light does not directly affect LiDAR returns — the sensor supplies its own illumination — but it matters at the **stack level** because cross-sensor fusion in [[7_0_overview_EN|Ch 7]] depends on camera information that low light degrades. The chapter has handled the per-stage half — §5.1's de-noising and intensity caveats, §5.2's ground-segmentation failure modes, §5.3's clustering breakdowns under sparse far returns. The deployment half is operational: validation sets must include adverse-weather rosbags, freshness monitors must detect sensor health degrading faster than the planner can react, and the Generic Obstacle Detection fallback on the ungated occupancy grid must stay active so a DL primary detector that fails on weather-distorted classes still has a class-agnostic backstop. Formal hazard analysis — fault trees, risk acceptance, residual-risk argument — belongs to [[11_0_overview_EN|Ch 11]]; this section catalogs the runtime symptoms.
